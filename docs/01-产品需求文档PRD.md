# Text-to-SQL 产品需求文档 (PRD)

## 1. 产品概述

### 1.1 产品定位
一款基于自然语言处理的智能数据库查询工具，用户可以通过自然语言描述需求，系统自动转换为SQL语句并在PostgreSQL/SQLite数据库中执行，返回真实数据。

### 1.2 核心价值
- **降低门槛**: 无需掌握SQL语法，任意用户都能进行数据库查询
- **提升效率**: 快速从数据中获取洞察
- **安全可控**: 内置查询验证和过滤机制

### 1.3 目标用户
| 用户类型 | 使用场景 | 痛点 |
|---------|---------|-----|
| 产品经理 | 数据分析、报表需求 | 不懂SQL，依赖开发 |
| 运营人员 | 业务数据查询 | 查询效率低 |
| 业务人员 | 日常数据获取 | 无法自主取数 |
| 初级开发者 | 学习SQL | 语法不熟悉 |

### 1.4 技术栈
| 层级 | 技术选型 |
|-----|---------|
| 后端框架 | .NET 8 / ASP.NET Core |
| API文档 | Swagger/OpenAPI |
| 数据库支持 | PostgreSQL, SQLite |
| AI模型 | Anthropic格式 (Claude API兼容) |
| 缓存 | Redis |
| 日志 | Serilog |

## 2. 功能需求

### 2.1 核心功能 (P0 - 必须)

#### F1: 自然语言转SQL
- 用户输入自然语言描述
- AI模型自动转换为SQL语句
- 支持SELECT查询
- 返回SQL语句和执行结果

#### F2: 数据库连接管理
- 支持PostgreSQL连接
- 支持SQLite连接
- 连接信息加密存储
- 支持连接测试

#### F3: SQL执行引擎
- SQL语法验证
- 执行超时控制
- 查询结果分页返回
- 支持CSV/Excel导出

#### F4: API接口
- Swagger UI页面
- RESTful API设计
- JWT认证
- 请求限流

### 2.2 增强功能 (P1 - 重要)

#### F5: SQL优化建议
- SQL语法优化
- 性能优化建议
- 索引建议

#### F6: SQL转自然语言
- 解释SQL含义
- 结果解读

#### F7: AI会话管理
- 多轮对话上下文
- 会话历史记录
- 会话重置

#### F8: 数据干扰过滤
- 过滤用户输入中的干扰信息
- 最佳实践实现
- 意图识别增强

### 2.3 扩展功能 (P2 - 期望)

#### F9: 多数据库支持
- MySQL
- SQL Server
- 其他数据库扩展

#### F10: 高级导出
- Excel多Sheet导出
- 自定义字段映射
- 定时任务导出

## 3. 竞品分析

### 3.1 竞品列表

| 产品 | 优点 | 缺点 | 差异化机会 |
|-----|-----|-----|-----------|
| **Chat2DB** | 开源、功能全面、支持多数据库 | AI功能较弱、部署复杂 | 专注AI体验、轻量部署 |
| **AI2SQL** | AI专注度高、界面简洁 | 收费、不支持自部署 | 开源+自部署+免费 |
| **Vanna AI** | 开源、可本地部署 | 需配置LLM、使用门槛高 | 一键部署、开箱即用 |
| **AskYourDatabase** | 集成度高 | SaaS模式、数据外传 | 私有化、数据安全 |
| **Supabase** | 生态完整 | 定位不同、非专业SQL工具 | 垂直场景优化 |

### 3.2 差异化策略

```
┌─────────────────────────────────────────────────────────┐
│                    本产品差异化定位                       │
├─────────────────────────────────────────────────────────┤

│  ✓ Anthropic支持 - 支持Claude等先进模型                   │
```

## 4. 用户故事 (User Stories)

### 4.1 核心用户故事

```
US-001: 作为业务人员，我想通过自然语言查询数据
  场景: "查一下今天注册的新用户有多少"
  期望: 系统返回正确的SQL和查询结果

US-002: 作为产品经理，我想导出数据到Excel
  场景: 点击导出按钮，下载包含查询结果的Excel
  期望: Excel格式正确，数据完整

US-003: 作为管理员，我想管理数据库连接
  场景: 在设置页面添加/编辑/删除数据库连接
  期望: 连接信息加密存储，可测试连接

US-004: 作为用户，我想查看历史查询
  场景: 查看之前的查询记录和结果
  期望: 可追溯、可复用

US-005: 作为开发者，我想通过API集成
  场景: 使用API_KEY调用查询接口
  期望: RESTful接口，JWT认证
```

### 4.2 边界场景

```
BE-001: 恶意SQL注入
  处理: 过滤DROP/DELETE/INSERT/UPDATE等写操作

BE-002: 超时查询
  处理: 30秒超时控制，防止慢查询

BE-003: 空结果查询
  处理: 返回空数据集，提示用户调整查询

BE-004: 无效自然语言
  处理: 返回友好的错误提示

BE-005: 大结果集导出
  处理: 分页导出，限制单次导出量
```

## 5. 非功能性需求

### 5.1 性能指标

| 指标 | 目标值 | 说明 |
|-----|-------|-----|
| API响应时间 | P95 < 500ms | 不含AI调用 |
| AI转SQL时间 | < 3s | 网络正常情况 |
| 并发支持 | 100+ | 支持多用户 |
| 数据导出 | 支持10000行 | Excel格式 |

### 5.2 安全要求

| 项目 | 要求 |
|-----|-----|
| 认证 | JWT Bearer Token |
| 加密 | AES-256-GCM |
| 限流 | 100请求/分钟 |
| SQL防护 | 禁止写操作+DML过滤 |

### 5.3 可用性

| 项目 | 要求 |
|-----|-----|
| 健康检查 | /health, /health/ready, /health/live |
| 日志 | 结构化日志 (Serilog) |
| 监控 | Prometheus格式指标 |
